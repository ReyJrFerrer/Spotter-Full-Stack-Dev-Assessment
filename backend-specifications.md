**Project Description**
This project is a full-stack web application built with Django and React that serves as a route and Electronic Logging Device (ELD) log generator for property-carrying commercial truck drivers. The application accepts four specific trip inputs: current location, pickup location, dropoff location, and current cycle hours used. Utilizing a free Map API, the system calculates an optimized route and strictly enforces Federal Motor Carrier Safety Administration (FMCSA) Hours of Service (HOS) regulations to automatically schedule mandatory stops, rests, and fueling. Finally, the system generates step-by-step route instructions and dynamically draws 24-hour ELD log sheets to visualize the driver's schedule.

***

### **Backend Specifications**

**1. Core System & Environment Architecture**
*   **Framework Integration:** The backend must be constructed using Python's Django web framework and the Django REST Framework (DRF) to serve a decoupled API to the React frontend.
*   **Cross-Origin Resource Sharing:** The `django-cors-headers` library must be configured to whitelist the frontend application, ensuring API requests between the client and server are not blocked.
*   **Directory Structure Strategy:** Utilize a modular Django app structure to isolate the routing and HOS logic engines from core settings.
*   **Containerization:** The backend should be structured to support Docker configuration for production deployment.

**2. API Endpoint Specifications**
*   **Primary Action Endpoint:** Implement a DRF view or ViewSet to handle `POST` requests for trip generation.
*   **Expected Request Payload (Inputs):** The API must strictly accept and validate the four required inputs: `current_location`, `pickup_location`, `dropoff_location`, and `current_cycle_used_hrs`.
*   **Expected Response Payload (Outputs):** The API must return the calculated route map coordinates, an array of scheduled stops and rests, and the daily log sheet data objects (handling multiple sheets if the trip exceeds 24 hours).

**3. External API Integration Requirements**
*   **Routing Execution:** The backend must integrate a free Map API to calculate driving distance, route paths, and estimated driving times between the inputted locations.

**4. Algorithmic Business Logic (FMCSA Constraints Playbook)**
The core algorithm must iteratively process the route fetched from the Map API and dynamically inject status changes based strictly on the following constraints:
*   **Duty Status Initialization:** The algorithm must categorize all time into one of four statuses: Off Duty, Sleeper Berth, Driving, and On Duty (Not Driving).
*   **Hardcoded Task Times:** The algorithm must explicitly inject 1 hour of "On Duty (Not Driving)" status for the pickup event, and 1 hour of "On Duty (Not Driving)" for the drop-off event.
*   **Fueling Mandate:** The algorithm must force a fueling stop at least once every 1,000 miles.
*   **The 11-Hour Driving Limit:** The algorithm must cap driving at 11 hours, requiring 10 consecutive hours off duty before driving can resume.
*   **The 14-Hour Window:** The algorithm must enforce a 14-consecutive-hour window that begins when the driver comes on duty; after 14 hours, no driving can be scheduled until a 10-hour off-duty reset is completed.
*   **The 30-Minute Rest Break:** The algorithm must inject a consecutive 30-minute break after 8 cumulative hours of driving.
*   **The 70-Hour/8-Day Limit Check:** The algorithm must track the `current_cycle_used_hrs` input. All new Driving and On-Duty time generated by the trip must be added to this total, and driving must be halted if the sum reaches 70 hours.
*   **Scope Exemptions:** The algorithm must assume there are no adverse driving conditions and the driver is hauling property.

**5. ELD Log Data Generation Engine**
The backend must format the chronological trip timeline into specific 24-hour data chunks to be rendered by the React frontend:
*   **24-Hour Slicing:** The timeline must be split precisely at midnight into multiple daily log objects if the trip spans multiple days.
*   **Grid Coordinate Mapping:** Generate start and end times for each continuous block of duty status so the frontend can draw accurate horizontal and vertical lines on the log's 15-minute increment grid.
*   **Required Remarks Generation:** Whenever a duty status change is triggered, the backend must generate a "Remarks" string containing the location where the change occurred, along with the specific activity.
*   **Header & Calculation Totals:** Each daily log data object must compute the total hours spent in each of the 4 duty statuses (summing to exactly 24 hours per day) and the total miles driven for that period.
